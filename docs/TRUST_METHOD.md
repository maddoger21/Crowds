# Метод оценки достоверности пользовательских геоданных

## Назначение

Метод предназначен для автоматической предварительной оценки сообщений пользователей о чрезвычайных и бытовых ситуациях. Он не заменяет администратора, но помогает выделять сообщения, которым можно доверять больше, и сообщения, требующие дополнительной проверки.

## Учитываемые признаки

Оценка достоверности учитывает:

- текущий статус сообщения: `PENDING`, `APPROVED`, `REJECTED`, `ARCHIVED`;
- количество подтверждений от других пользователей;
- количество жалоб на недостоверность или нарушение;
- число возможных дублей события рядом по месту и времени;
- базовую репутацию автора;
- давность сообщения.

## Формула trustScore

Расчёт вынесен в `TrustScoreCalculator`, чтобы его можно было тестировать отдельно от Android и Firebase.

Базовая оценка по статусу:

- `PENDING` = 40;
- `APPROVED` = 60;
- `REJECTED` = 10;
- `ARCHIVED` = 30.

Дополнительные компоненты:

- `confirmationScore = min(confirmationsCount * 5, 25)`;
- `reportPenalty = min(reportsCount * 10, 35)`;
- `duplicatePenalty = min(duplicateCount * 10, 20)`;
- `authorScore = authorReputation.coerceIn(0.0, 20.0)`.

Штраф за давность:

- меньше 24 часов: 0;
- от 24 до 72 часов: 5;
- больше 72 часов: 10.

Итоговая формула:

```text
score = baseScore + confirmationScore + authorScore
        - reportPenalty - duplicatePenalty - recencyPenalty
```

Результат ограничивается диапазоном `0..100`.

Уровни достоверности:

- `score >= 75`: высокая;
- `score >= 50`: средняя;
- `score >= 30`: низкая;
- `score < 30`: сомнительная.

## Подтверждения

Пользователь может подтвердить чужое сообщение один раз. Подтверждение хранится в `posts/{postId}/confirmations/{uid}`. При подтверждении транзакция Firestore:

- создаёт документ подтверждения;
- увеличивает `confirmationsCount`;
- пересчитывает `trustScore` и `trustLevel`;
- увеличивает `users/{uid}.confirmationsMade`.

Автор сообщения не может подтверждать собственный пост.

## Жалобы

Пользователь может пожаловаться на чужое сообщение один раз. Жалоба хранится в `posts/{postId}/reports/{uid}` и содержит причину. При жалобе транзакция Firestore:

- создаёт документ жалобы;
- увеличивает `reportsCount`;
- пересчитывает `trustScore` и `trustLevel`;
- увеличивает `users/{uid}.reportsMade`.

Если жалоб становится 3 или больше, либо оценка падает ниже 30, пост получает `needsAdminReview = true`.

## Пространственно-временная проверка дублей

Перед созданием нового поста приложение загружает сообщения за последние 3 часа и сравнивает координаты. Если расстояние до существующего сообщения меньше 0.2 км, оно считается возможным дублем.

Найденные идентификаторы сохраняются в `duplicateCandidateIds`, а их количество в `duplicateCount`. Пользователь видит предупреждение и может отказаться от создания нового сообщения.

## Использование администратором

Администратор видит сообщения:

- со статусом `PENDING`;
- с `needsAdminReview = true`;
- с `trustScore < 30`;
- с `reportsCount >= 3`.

В карточке события администратор видит оценку достоверности, количество подтверждений и жалоб, число возможных дублей. Он может одобрить, отклонить, архивировать или удалить сообщение. При изменении статуса `trustScore` пересчитывается.

## Ограничения

Метод является эвристическим. Он не гарантирует абсолютную истинность сообщения и должен использоваться как инструмент поддержки модерации. Базовая репутация реализована просто и не учитывает долгосрочную историю качества сообщений. Проверка дублей использует только расстояние и окно времени, без анализа текста описания или категории события.
